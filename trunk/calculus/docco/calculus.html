Further Work

Big graphs and the monitor

The need_refresh function in the trad4 internals explicidly lists each object to which any type subscribes. This has worked well until now, but presents a problem with the monito and big graphs. As the monito subscribes to 3*NUM_NODES object (as there are 3 tiers), big graphs, for example NUM_NODES=10k, mean a massive need_refresh function as 30k objects are listed in this function. This means for a long compile time and a big binary - monitor.o is 3meg. This doesn't actually affect run-time as because eveery object to which the monitor subscribes will have changed, lazy evaluation kicks in so only the first object needs to be tested.
